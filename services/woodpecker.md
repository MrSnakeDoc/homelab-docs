# Woodpecker

Woodpecker is the continuous integration and automation platform of the homelab.

It automatically builds, tests, and processes source code repositories hosted on the internal Git platform. Pipelines are triggered by repository events and executed in isolated build environments.

Woodpecker forms the **automation layer** of the development platform.

---

## Purpose

Woodpecker exists to automate development workflows.

Its primary responsibilities include:

* executing build pipelines
* running automated tests
* building artifacts
* validating code changes
* coordinating automated tasks triggered by repository events

Pipelines are defined directly in repositories using a `.woodpecker.yml` configuration file.

Woodpecker does not perform:

* source code hosting
* repository management
* version control
* long-term artifact storage

Those responsibilities belong to the Git platform and other infrastructure services.

---

## Architectural Role

Woodpecker represents the **automation and build execution layer** of the development environment.

Repository events generated by the Git platform trigger automation pipelines.

```text
Developer
   → Git push
      → Git platform
         → Webhook
            → Woodpecker
```

Woodpecker evaluates the pipeline definition stored in the repository and schedules build jobs accordingly.

---

## Pipeline Execution Model

Woodpecker follows a **server and runner architecture**.

The central Woodpecker server manages:

* pipeline orchestration
* repository synchronization
* build scheduling
* pipeline status reporting

Build execution itself is delegated to worker nodes called **runners**.

```text
Woodpecker server
   → Runner
      → Build container
```

Each runner executes build steps inside isolated containers.

This separation allows pipelines to run without granting the main server direct access to build environments.

---

## Repository Integration

Woodpecker integrates directly with the Git platform.

Repository events such as:

* pushes
* pull requests
* tag creation

can trigger pipeline execution.

```text
Repository event
   → Git platform
      → Webhook
         → Woodpecker
            → Pipeline execution
```

This model ensures that automation remains tightly coupled with the repository lifecycle.

---

## Deployment Model

The CI platform runs as a containerized service within the application services layer.

Key characteristics include:

* centralized pipeline orchestration server
* multiple build runners
* container-based build execution
* persistent storage for pipeline state and logs
* HTTPS access through the reverse proxy layer

Runners operate independently and connect to the server using a shared authentication mechanism.

---

## Runner Model

Runners execute pipeline steps using containerized workloads.

Each runner:

* connects to the central Woodpecker server
* receives scheduled build jobs
* executes build steps inside containers
* reports execution results back to the server

Multiple runners allow concurrent build execution and horizontal scalability.

```text
Woodpecker server
   → Runner 1
   → Runner 2
   → Runner N
```

The number of runners determines the overall build capacity of the system.

---

## Access Model

The Woodpecker web interface allows administrators and developers to observe pipeline activity.

```text
Browser
  → Internal DNS
    → Reverse proxy
      → Woodpecker
```

The interface provides:

* pipeline status
* build logs
* repository integration status
* execution history

Pipeline execution itself occurs entirely within the runner environment.

---

## Authentication Model

Woodpecker relies on the Git platform for authentication and repository ownership information.

Users authenticate through their Git accounts and permissions are derived from repository access rights.

This ensures that only authorized developers can trigger or manage pipelines.

---

## Failure Scenarios

| Failure                       | Impact                        |
| ----------------------------- | ----------------------------- |
| Woodpecker server unavailable | Pipelines cannot be scheduled |
| Runner unavailable            | Build capacity reduced        |
| All runners unavailable       | Pipelines remain queued       |
| Git platform unavailable      | No pipeline triggers          |

Failures impact development automation but do not affect infrastructure services.

---

## Security Model

The CI system follows several security principles:

* build execution isolated inside containers
* runners separated from the orchestration server
* no direct repository access outside Git authentication
* HTTPS enforced for web interface access
* containerized execution environments

This model reduces the risk of build pipelines affecting the underlying infrastructure.

---

## Design Principles

The CI platform follows several operational principles:

* automation triggered by repository events
* clear separation between orchestration and execution
* containerized build environments
* scalable runner architecture
* minimal coupling with infrastructure services

Woodpecker acts as the **automation engine** of the internal development platform.
